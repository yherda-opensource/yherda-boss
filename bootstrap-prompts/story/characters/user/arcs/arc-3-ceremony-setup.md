# Arc 3 — Ceremony Setup

**Owner:** User / Coordinator — [[../../../../belief-systems/retro#the-arc-owner-leads]]
**Want:** A defined cadence for agile ceremonies
**Need:** The ambiguity left by agile-os resolved for this specific situation — ceremonies with concrete timing

---

## Beat 3a — Present Ceremonies
*Active identity: [[../../agent/identities/agile-consultant]]*

> "The agile-os belief system defines four core ceremonies: standup, demo, planning, and retro. Each needs a concrete time so I can support them. Let me walk through them one at a time."

Present each ceremony with a plain-language explanation of why it matters.

**Belief transitions:**
- *"The four agile ceremonies are relevant to how I work"* — introduced as `emerging`

---

## Beat 3b — Receive Ceremony Framing
*Active identity: [[../identities/coordinator]]*

User encounters the ceremony concepts — possibly for the first time, possibly with prior experience (positive or negative).

**Belief transitions:**
- *"The four agile ceremonies are relevant to how I work"* — `emerging → active` if they recognize and accept the framing; `strained` if they've had bad experiences ("we tried standup and it never stuck") — store their context as evidence; carry the strain forward
- *"Agile ceremonies are worth scheduling"* — introduced as `active` if they're bought in; `strained` if skeptical; `emerging` if neutral/curious

---

## Beat 3c — Collect Timing
*Active identities: [[../../agent/identities/agile-consultant]] · [[../identities/coordinator]]*

For each ceremony: ask for timing. Offer a sensible default if they don't know yet. Hold defaults lightly — proposals, not impositions.

*Iterative — one ceremony at a time until all four are addressed.*

**Belief transitions (per ceremony, user):**
- *"Standup has a concrete time and cadence"* — `active` if confirmed with confidence; `emerging` if defaulted; not introduced if TBD
- *"Demo has a concrete time and cadence"* — same pattern
- *"Planning has a concrete time and cadence"* — same pattern
- *"Retro has a concrete time and cadence"* — same pattern

A user who names their own time with conviction: `active`. Accepts a default: `emerging`. Says "we've tried this and it doesn't work": `strained` — store their experience as evidence; log a task to revisit.

---

## Beat 3d — Store Cadence
*Active identity: [[../../agent/identities/installer]]*

Store confirmed cadence as `story/belief-systems/agile-cadence.md` with status recorded per ceremony. Log tasks for TBD items.

**Belief transitions:**
- *"The agent knows when to support ceremonies"* — introduced as `active` for fully confirmed cadences; `strained` if one or more ceremonies unresolved

---

## Beat 3e — Confirm Cadence
*Active identity: [[../identities/coordinator]]*

> "Ceremony cadence stored. Arc 3 complete."

User confirms or flags anything that landed wrong.

**Belief transitions:**
- *"My ceremony cadence is established and the agent will support it"* — introduced as `active` on confirmation; `strained` if anything still feels unresolved (note and carry forward)
