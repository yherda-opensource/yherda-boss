# Belief System — Agile Setup

**Version:** 0.2 (draft)
**Status:** Emerging
**Tags:** #belief-system #bootstrap #agile-setup

---

## Prime Directive

The beliefs required to instantiate agile concepts for a specific user without imposing a template. Every agile arc begins with a retro on what the user currently believes about how they work — not with what agile says they should believe.

---

## Core Beliefs

### 1. Retro first, introduce second
Before presenting any agile concept, ask what the user currently believes about that domain. Establish their existing belief status before introducing agile beliefs alongside them. The reconciliation happens in the open — not by overwriting what they already hold.

### 2. Prior bad agile experience is evidence, not an objection
When a user says "we tried standup and it never stuck," that is a `strained` belief with real evidence behind it. Acknowledge it as evidence. Store it. Find out what failed and why — that shapes what gets instantiated differently here.

### 3. Defaults are proposals, not impositions
Base DoR, DoD, and ceremony timing are starting points. The user's context overrides them. A default accepted without pushback arrives as `emerging` — it hasn't been tested yet. A belief stated with conviction from experience arrives as `active`.

### 4. Minimum viable cadence over ideal cadence
Don't demand a full agile ceremony set. Establish what the user can actually commit to — a review cycle, a planning moment, a reflection habit. The goal is a belief about how often to review and adjust, and a planning cycle if one exists. Everything else is optional until it's needed.

### 5. Abstract agile concepts need concrete instantiation to be load-bearing
"We have a Definition of Done" is not a belief. "Code is done when tests pass, documentation is updated, and a stakeholder has confirmed it meets the need" is a belief. An arc is complete when abstractions have become specific enough to act on.

### 6. Initial and ideal diverge intentionally on agile beliefs
The initial workspace carries the user's actual agile practice — including strained beliefs about ceremonies they've tried and abandoned. The ideal workspace carries the clean agile belief set. The divergence between them in the agile domain is expected and valuable — it shows exactly where the friction with agile lives for this user.

### 7. Guardrails that can't be wired become tasks
Anything the agent can't wire directly in the conversation gets logged as a task — deferred fidelity, not dropped.

### 8. The working agreement is a belief system, not a policy document
Working agreements stored as beliefs with statuses are things the agent can reason from and update through retro. Working agreements that live in a doc nobody reads are aspirations, not beliefs.
